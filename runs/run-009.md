## Pre-arm mechanism

When `POST /collect` fires, the coordinator transitions from IDLE → COLLECTING. At that moment,
no executor cloudagent is connected (only `otel-converter-bdccdc10`). When the executor pod's
cloudagent connects later (after pip-install completes), the coordinator sends START to it
immediately on join — because a collection is already ACTIVE.

This is visible in the early pod logs:
```
[Nsight INFO] Current session: f9744bd9-0a53-4c59-9f8f-71b9b2b73e89
[Nsight INFO] Current collection: 14e4aea5-aa27-49da-8477-23162d7ed95a
```

These lines appear BEFORE `python3 -m kfp.dsl.executor_main` starts its CUDA work, confirming
the cloudagent received START before CUDA context initialization.

## Dropped events analysis

`Number of incomplete CUPTI events dropped: 6,025` — the hardware trace buffers contain
KERNEL records for all ~6,051 iterations, but 6,025 had incomplete GPU timestamp data at
stop time. The `STOP_TRIGGER=Delayed` (process exit) gives nsys limited time to flush hardware
counters; the final buffer is not fully drained.

**27 + 6,025 ≈ 6,052 total captured kernel records** — confirming hardware trace IS recording
all iterations, not just the 1.89s startup snapshot.

To capture all kernel records (not just 27), the collection needs to be stopped BEFORE the
process exits, giving the hardware trace buffers time to flush. This could be accomplished with:
1. A longer post-CUDA sleep (currently 60s) — may already be enough to flush most buffers
2. DELETE /collect while the post-CUDA sleep is running
3. Adding `cudaDeviceSynchronize()` before process exit (already present as loop body)
