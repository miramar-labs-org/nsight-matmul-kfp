## What changed from run-005

Added `--env-var NVIDIA_VISIBLE_DEVICES=all` to nsightToolArgs. The intent was to override the
hook's anti-recursion guard (`NVIDIA_VISIBLE_DEVICES=void` in PROCESS_0's env), hypothesizing
that "void" was causing `COLLECT_CUDA_TRACE=false`.

Both hypotheses were **wrong**:
1. `COLLECT_CUDA_TRACE=false` is the expected state in process hook injection mode (not caused
   by NVIDIA_VISIBLE_DEVICES). It persisted in run-006 even after the override.
2. `--env-var` affects PROCESS_0's environment, not nsys's own environment.
   TARGET_INFO_SYSTEM_ENV (nsys's env) still shows `NVIDIA_VISIBLE_DEVICES=void` in run-006.

Additionally, in this run the coordinator was used properly (POST /collect sent), resulting in
`START_TRIGGER=Manual` instead of `START_TRIGGER=Delayed` (run-005). The Manual trigger
disrupted the hardware trace snapshot timing.

---

## Root cause of regression (from KFP logs)

The KFP logs reveal what actually happened:

```
sleeping 60s (nsys init window) ...
[Nsight INFO] Coordinator Session ... Begun
[Nsight INFO] START collection ...       ← auto-fired immediately by coordinator
[Nsight INFO] Collection Starting
starting CUDA matmul loop (180s) ...
iters=1000 elapsed=38.6s
[Nsight INFO] STOP collection            ← coordinator cuts off at ~98.5s total
```

The `nsight-coordinator-cloudagent` injects `--start-later=true` into the nsys command and
**auto-fires START when the pod is detected** — no manual POST /collect needed. This is the
`START_TRIGGER=Manual` state, but it's triggered automatically, not by a human.

**The real killer: the 60s pre-CUDA sleep consumed 60s of the ~98.5s session window.**

- Coordinator session window: ~98.5s from nsys start  
- Pre-CUDA sleep: 60s wasted  
- Remaining for CUDA work: 98.5 - 60 = **38.5s**  
- Result: collection stopped at `iters=1000` (38.6s into the 180s matmul loop)

The `--env-var NVIDIA_VISIBLE_DEVICES=all` was an additional wrong change — it did not affect
COLLECT_CUDA_TRACE and disrupted hardware trace snapshot timing (47→2 kernels).

## Diagnostics

- `PROCESS_0:COLLECT_CUDA_TRACE=false` — same as run-005; not fixed by `--env-var`
- `NVIDIA_VISIBLE_DEVICES=void` in TARGET_INFO_SYSTEM_ENV — nsys's own env is still void;
  `--env-var` only changes PROCESS_0's env, not nsys's
- `START_TRIGGER=Manual, STOP_TRIGGER=Delayed` — coordinator triggered start; collection
  stopped after 98.5s when nsys exited
- Hardware trace captured only 2 randn kernels; ZERO matmul kernels despite 40s of matmul

---

## Key findings for future runs

- **DO NOT use `--env-var NVIDIA_VISIBLE_DEVICES=all`** — it reduces hardware trace coverage
  from 47 to 2 kernels
- **Avoid Manual trigger (POST /collect)** — the Delayed trigger (no coordinator start signal)
  gives better hardware trace coverage (47 vs 2 kernels)
- `COLLECT_CUDA_TRACE=false` is expected and does NOT mean GPU kernels aren't captured —
  it means hardware trace is used instead of software CUPTI; hardware trace is a single ~1.89s
  snapshot at CUDA init time
