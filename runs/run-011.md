## Root cause of 0 kernel records

**Hardware trace relies on GPU context destruction for its ring buffer flush.**

When `DELETE /collect` fires during the 60s sleep:
1. nsys receives STOP signal while python3 is still alive (in `time.sleep(60)`)
2. nsys finalizes the report immediately — reads what's in the hardware counter ring buffer
3. The ring buffer has 5,516 events with incomplete GPU timestamps (the GPU hasn't finished
   the timestamp pipeline for the last ~5,515 kernel dispatches)
4. All 5,516 events are dropped as incomplete → 0 kernel records

When `STOP_TRIGGER=Delayed` (process exits naturally):
1. python3 exits → PyTorch/CUDA runtime destroys the GPU context
2. GPU context destruction forces a full hardware trace ring buffer flush
3. Some events get their timestamps committed before the flush completes
4. 27–166 kernel records survive (with the rest still dropped as incomplete)

**The 60s post-CUDA sleep with Delayed stop (run-010: 166 kernels)** is better than
**DELETE during the sleep (run-011: 0 kernels)** because the sleep allows the hardware
trace ring buffer to drain via the flush timer (`--cuda-flush-interval=100`) BEFORE
the GPU context is destroyed at process exit.

## Key findings summary

The hardware trace snapshot approach (`COLLECT_CUDA_TRACE=false`, hardware mode) on GB10
has a fundamental coverage limitation:

1. Collection must start before CUDA init (pre-arm works, run-009/010/011 all confirmed)
2. Stop must happen AT or AFTER GPU context destruction (process exit = best)
3. The 60s post-CUDA sleep + Delayed stop gives ~166 kernel records (best so far)
4. Firing DELETE before process exit drops all remaining hardware events (0 kernels)

**Estimated maximum recoverable with current approach:** ~166 out of ~5,500+ kernel launches —
approximately 3% coverage. The hardware trace ring buffer holds a finite number of events;
the rest are dropped regardless of flush timing.

## What to try next (if full kernel coverage is needed)

**Option A — CUPTI software trace:** Switch from hardware trace to CUPTI software trace by
setting `NVIDIA_VISIBLE_DEVICES` to the actual GPU index (not `void`) in the process hook
env. This requires modifying how `libNsightProcessHook.so` sets up the child process, or
setting `COLLECT_CUDA_TRACE=true` explicitly. Would give 100% kernel coverage.

**Option B — NVTX + API trace only:** The current 100% CUDA API trace (all `cudaDeviceSynchronize`,
`cuLaunchKernel` calls) is complete. For the matmul microbenchmark, this is sufficient:
cuBLAS kernel execution time = `cudaDeviceSynchronize` time (~29ms avg). No hardware trace needed.

**Option C — Longer workload with smaller matrix:** Reduce matrix size so more iterations
fit in the hardware trace snapshot window (currently ~1.89s at CUDA init).
