## What changed from run-006

- Removed the 60s pre-CUDA sleep from the component. KFP logs for run-006 showed that the
  coordinator auto-fired START before CUDA init even with the sleep; the sleep was consuming ~60s
  of the ~98.5s session window (reducing captured CUDA time from 180s to ~38.5s).
- values.yaml reverted: removed `--env-var NVIDIA_VISIBLE_DEVICES=all` (which had caused the
  hardware trace to regress from 47 → 2 kernels in run-006).

---

## What happened

The coordinator session `f9744bd9-0a53-4c59-9f8f-71b9b2b73e89` remained **IDLE** for the entire
run. No collection was triggered, no profiling data was captured, and no report appeared in MinIO.

---

## Root cause

Session auto-rotation — the mechanism that had been creating new sessions and auto-firing START
in run-004, run-005, and run-006 — stopped working after the Helm upgrade that deployed the
reverted values.yaml (`be5eccb`). The coordinator retained the existing session `f9744bd9` but
stopped creating new sessions for new pods.

In the auto-rotation model:
1. A new pod appears → coordinator creates a new session
2. New session → auto-trigger START when executor cloudagent connects
3. START arrives at cloudagent before CUDA init → hardware trace fires

Without session rotation, `f9744bd9` stayed IDLE with no auto-trigger. The executor cloudagent
connected, received no START command, and nsys collected nothing.

Evidence: `GET /api/v1/sessions/` after the run showed `f9744bd9` still in IDLE state with
`activeCollectionID: null`.

---

## Learning

Manual `POST /api/v1/sessions/{sid}/collect` is required to trigger collection when
auto-rotation is not occurring. This was confirmed by run-008 (manual POST mid-run) and
run-009 (manual POST pre-deploy).
