## How GPU-side tracing works in process hook injection mode

The Nsight Operator intercepts `python3` via LD_PRELOAD (process hook). It launches nsys as the
profiler, which then relaunches python3 as `PROCESS_0`. In this injection topology:

- `PROCESS_0:COLLECT_CUDA_TRACE=false` ŌĆö expected for process hook injection mode. This flag
  indicates the standard CUPTI software activity path (nsys-owned CUDA context) is not used.
  Instead, nsys uses the GPU **hardware trace buffer** for GPU-side kernel records.
- `START_TRIGGER=Delayed` ŌĆö nsys started collecting immediately on process launch (no coordinator
  "start" signal sent). This allowed the hardware trace to arm at CUDA init time.
- The hardware trace on GB10 captures a **single ~1.89-second snapshot** of ~45 kernel events
  at the start of CUDA activity. After the snapshot buffer fills, no additional GPU-side kernel
  records are captured for the remainder of the session.
- CPU-side CUDA API events (CUPTI_ACTIVITY_KIND_RUNTIME) and NVTX events are captured
  **continuously** for the full collection window (79 seconds in this run).
