## Coordinator REST API (key finding)

The coordinator exposes a REST API at port 80 of the `nsight-operator-coordinator` service:

```
POST   /api/v1/sessions/{sid}/collect   → start collection (returns collection ID + clients)
DELETE /api/v1/sessions/{sid}/collect   → stop collection
GET    /api/v1/sessions/{sid}           → session status (IDLE / COLLECTING / ENDED)
POST   /api/v1/sessions/               → create a new session (BeginMessage: {tag, title})
```

No auth required in our cluster setup (CURVE encryption on ZMQ, HTTP open).

## Trigger timing problem

POST /collect was sent at **iters=1600 elapsed=58.3s** into the 180s CUDA loop. Result:

| Metric | Value |
|--------|-------|
| START_TRIGGER | Manual |
| STOP_TRIGGER | Delayed (process exit) |
| RUN_DURATION_MS | 180,506 (full CUDA loop) |
| CUPTI_ACTIVITY_KIND_KERNEL | ❌ MISSING (hardware trace snapshot not captured) |
| CUPTI_ACTIVITY_KIND_RUNTIME | ✅ 10,530 events over **119.9s** |
| NVTX events | ✅ 6,998 (3,499 bench push/pop pairs) |

Hardware trace snapshot missed: the 1.89s hardware trace buffer fires at the **first CUDA kernel
after collection start**. Since collection started at iters=1600 (58s into CUDA loop), the CUDA
context was already initialized and the initial warmup had already passed. In Manual trigger mode,
the hardware trace does not retroactively capture earlier kernels.

## Strategy for run-009: pre-deploy POST /collect

To capture the hardware trace snapshot, `POST /collect` must be sent BEFORE CUDA init. Since the
coordinator allows starting a collection even when no cloudagents are connected, the plan is:

1. POST /collect to session `f9744bd9` **before** deploying the KFP pod
2. Session transitions: IDLE → COLLECTING (collection ID pre-created)
3. Deploy pod → pip install cloudagents connect → coordinator sends START to them (no CUDA, no harm)
4. pip install cloudagents disconnect
5. Executor cloudagent connects → coordinator sends START (or cloudagent receives active collection)
6. Executor does CUDA init → hardware trace arms → fires on first kernel (warmup)
7. Collection captures hardware trace + full RUNTIME + NVTX for the 180s run

If the coordinator replays START to newly joining cloudagents when a collection is ACTIVE, this will
automatically give hardware trace coverage equivalent to run-005 (START_TRIGGER=Delayed, 47 kernels).
