## What changed from run-003

Root cause of run-002/003 empty captures: **timing**. The matmul completed in under 1 second (PyTorch import
overhead consumed ~18s of the 19s pod lifetime), and the Nsight coordinator's GO signal arrived after the
CUDA work was already done (START_TRIGGER: Manual, coordinator round-trip ~19s).

Fix applied:
- Added 60s pre-CUDA sleep (gives nsys time to initialize and receive the coordinator GO signal)
- Changed to a 180s CUDA loop (wide capture window, ‚Čę coordinator round-trip)
- Added 60s post-CUDA sleep (nsys flush + MinIO upload before exit)
- Started Nsight collection AFTER `main` container was Running (not before deploy)

## Diagnostics / known issues

- `PROCESS_0:COLLECT_CUDA_TRACE=false` in SQLite metadata ‚ÄĒ **expected** in process hook injection
  mode. nsys uses GPU hardware trace (not software CUPTI) for GPU-side kernel records. The 7 kernel
  instances from `cuda_gpu_kern_sum` come from the hardware trace binary section of the .nsys-rep
  (not from a SQLite CUPTI_ACTIVITY_KIND_KERNEL table, which is absent in this run).
  The 2,741 `cudaDeviceSynchronize` calls in `cuda_api_sum` are CPU-side CUDA API events, which ARE
  captured continuously via CUPTI_ACTIVITY_KIND_RUNTIME.
- Hardware trace: 7 kernel instances over ~140s profiling window. This is a hardware trace snapshot;
  privileged=false at this point limited the snapshot to 7 records. (run-005 bumped to 47 with
  `privileged: true`.)
- `nsys exit code = 139` ‚ÄĒ segfault after report generation. `.nsys-rep` is fully uploaded to MinIO
  before the crash; data is not lost. Pre-existing issue on GB10/DGX Spark.
- `Unable to configure CPU IP/backtrace samples` ‚ÄĒ expected, seccompProfile still blocks perf_event_open
  for non-cuda traces. Does not affect CUDA capture.
